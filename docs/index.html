Fivetran · dbt Labs · MDLS

467 schemas, 163,450 tables, one destination.

chris_rudolph_gcs_mdls is a Fivetran Managed Data Lake Service destination fed by 467 identical SQL Server connectors. Each lands its own schema of 350 tables. Nothing downstream can query "all of it" without manually unioning 467 schemas by hand — this page documents the actual shape of that problem and the dbt Labs pattern that collapses it back down to something queryable.

02Why this goes sideways in production

The schemas are all shaped identically (ft_table_0001 ... ft_table_0350 in every one), which is exactly what makes manual handling of this dangerous at this scale:

  • Catalog bloat. 163,450 tables in one destination's information_schema makes metadata queries (and dbt's own catalog generation) noticeably slower.
  • No single query surface. An analyst asking "show me row counts across all 467 source databases" has to hand-write a 467-way UNION ALL, or more realistically, doesn't ask the question.
  • One-model-per-table doesn't scale. Hand-authoring 163,450 dbt models (or even 350 x 467 staging models) is not a real option — the models have to be generated from a pattern, not written by hand.
  • Governance sprawl. 467 schemas means 467 places a permission grant, a masking policy, or a retention rule can silently diverge.

03The consolidation pattern

SQL Server x467 (ft_scale_db_0001 ... ft_scale_db_0467) │ Fivetran (467 zzz_ connectors, service: sql_server) ▼ chris_rudolph_gcs_mdls (MDLS destination, 467 schemas x 350 tables) │ dbt Labs — dynamic relation discovery, not hand-written models ▼ transform/models/consolidated/ (350 unified models, one per table name) │ ▼ BI / agents / analysts — query one table, see all 467 sources

Two dbt_utils macros do the actual work: get_relations_by_pattern finds every relation across the 467 schemas that shares a given table name, and union_relations unions them into one relation, stamping each row with which source schema it came from.

{{ macro consolidate_scale_table(table_name) }}
  {%- set relations = dbt_utils.get_relations_by_pattern(
      schema_pattern='zzz_sql01_ft_scale_db_%',
      table_pattern=table_name
  ) -%}
  with unioned as (
    {{ dbt_utils.union_relations(relations=relations) }}
  )
  select *, regexp_extract(_dbt_source_relation,
    'zzz_sql01_ft_scale_db_(\d+)', 1) as source_scale_db_id
  from unioned
{{ endmacro }}

Every one of the 350 generated models is then a one-line call: {{ consolidate_scale_table('ft_table_0001') }}. A script (scripts/generate_consolidation_models.py) writes all 350 files — it's a five-minute regeneration if the source table count ever changes, not a rewrite.
