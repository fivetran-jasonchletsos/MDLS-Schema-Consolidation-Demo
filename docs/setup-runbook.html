Getting this repo running on your machine, step by step

You've run dbt Core before with vi as your editor, and everything else went through Fivetran. This repo works differently: the transformation logic lives in GitHub as code, you edit it locally in VS Code, and dbt Fusion runs it. Fivetran's job doesn't change — it still lands the 467 SQL Server schemas into chris_rudolph_gcs_mdls. What's new is everything downstream of that.

Every step below says what to click or type, then what's actually happening underneath it, then links to the real documentation if you want to go deeper than what's written here. Work through it at your own pace — ping Jason if a step doesn't match what's on your screen.

Step 1 of 16 · one-time machine setup

01Install Git

Do this

Open Terminal (Applications → Utilities → Terminal) and run:

git --version

On a Mac, if Git isn't installed yet, this command itself pops up a system dialog offering to install the Xcode Command Line Tools. Click Install and wait for it to finish, then re-run the command — you should see a version number like git version 2.4x.x.

What's happening: Git is the version-control tool that tracks every change to every file in this project. Everything else in this guide — cloning, committing, pull requests — is built on top of Git. It's separate from GitHub: Git is the tool on your machine, GitHub is the website that hosts a copy of the repo.
Reference
  • git-scm.com/downloads — official Git installer, if the Xcode prompt doesn't appear
Step 2 of 16 · one-time machine setup

02Authenticate with GitHub

Do this

Install the GitHub CLI (brew install gh, or download it from the link below), then in Terminal run:

gh auth login

Choose GitHub.com, then HTTPS, then answer yes when it asks to authenticate Git with your GitHub credentials. Follow the browser prompt to finish signing in.

What's happening: Cloning a private-ish or org-owned repo and pushing changes back both require GitHub to know it's really you. gh auth login handles that once, so every future git push / git pull just works without you typing a password or managing an SSH key by hand.
Reference
  • GitHub CLI quickstart — official GitHub Docs
  • gh auth login manual — full flag reference

Step 5 of 16 · one-time machine setup

05Install the dbt Fusion engine

Do this

If the extension didn't already prompt you to install it, run one of these in Terminal (macOS/Homebrew shown; other options are in the reference link):

brew install dbt

or:

curl -fsSL https://public.cdn.getdbt.com/fs/install/install.sh | sh -s -- --update

Confirm it worked:

What's happening: This is the actual engine that reads transform/dbt_project.yml and runs your models. 06Clone this repo — the literal answer to "how do I open a GitHub repo from the dbt Fusion interface"

Do this
  1. In VS Code, press Cmd+Shift+P to open the Command Palette.
  2. Type Git: Clone and select it, then choose Clone from GitHub.
  3. Paste this repo's URL when prompted (or search for it if you're signed in): 
    https://github.com/fivetran-jasonchletsos/MDLS-Schema-Consolidation-Demo.git
  4. Pick a local folder to store it — anywhere under your home folder is fine, e.g. ~/Documents/GitHub/.
  5. When VS Code asks, click Open to open the cloned repo in a new window.
What's happening: "Cloning" downloads a full working copy of this repo — every file, plus its entire history — onto your machine. This is the one step that replaces "running everything from Fivetran": from here on, the project lives on your laptop and on GitHub, and Git keeps the two in sync.

Step 8 of 16 · per-repo, do once for this project

08Set up your connection profile

Get these four things from Jason first — via 1Password, not Slack This destination connects through Snowflake, using key-pair authentication rather than a password. Jason will send you, out of band: the private key file (.p8), its passphrase, and the warehouse/database to use. None of that ever goes in this repo or in this doc — see why below.
Do this
  1. Save the .p8 key file Jason sends you somewhere outside this repo, e.g. ~/.ssh/mdls_demo_key.p8.
  2. Export the connection details as environment variables in your shell profile (~/.zshrc or ~/.bash_profile): 
    export SNOWFLAKE_ACCOUNT="..."
export SNOWFLAKE_USER="..."
export SNOWFLAKE_ROLE="..."
export SNOWFLAKE_WAREHOUSE="..."
export SNOWFLAKE_DATABASE="..."
export SNOWFLAKE_PRIVATE_KEY_PATH="$HOME/.ssh/mdls_demo_key.p8"
export SNOWFLAKE_PRIVATE_KEY_PASSPHRASE="..."
    Then reload your shell: source ~/.zshrc.
  3. Create (or open) ~/.dbt/profiles.yml — note the leading ~/, this lives in your home folder, outside the repo. Add a profile whose name matches this project's expected profile name exactly, referencing the env vars instead of literal values: 
    mdls_schema_consolidation:
  target: dev
  outputs:
    dev:
      type: snowflake
      account: "{{ env_var('SNOWFLAKE_ACCOUNT') }}"
      user: "{{ env_var('SNOWFLAKE_USER') }}"
      role: "{{ env_var('SNOWFLAKE_ROLE') }}"
      warehouse: "{{ env_var('SNOWFLAKE_WAREHOUSE') }}"
      database: "{{ env_var('SNOWFLAKE_DATABASE') }}"
      schema: consolidated
      private_key_path: "{{ env_var('SNOWFLAKE_PRIVATE_KEY_PATH') }}"
      private_key_passphrase: "{{ env_var('SNOWFLAKE_PRIVATE_KEY_PASSPHRASE') }}"
What's happening: transform/dbt_project.yml declares profile: 'mdls_schema_consolidation'. dbt looks for a profile with that exact name in ~/.dbt/profiles.yml to know which warehouse to connect to and how to authenticate. The env_var(...) calls mean the actual account, user, and key never get typed into a file that could end up in Git — profiles.yml only holds variable names, and the real values live in your shell environment, outside any repo. Key-pair auth means there's no password to type at all: dbt proves who you are using the private key file instead.

Step 12 of 16 · the actual run

12Run the project

Do this

From inside transform/:

dbt run --select consolidated

Or, from inside the VS Code extension: open any file in transform/models/consolidated/ and click the dbt icon in the editor toolbar to run just that model.

What's happening: This builds all 350 consolidated views in your warehouse. Each one unions the same-named table across all 467 source schemas and tags every row with source_scale_db_id so you can tell which of the 467 source databases it came from.

Step 14 of 16 · the ongoing loop, every time you make a change

14Make a change and send it back

Do this, in order, every time
  1. git pull — get the latest version before you start.
  2. Create a branch for your change (VS Code Source Control panel → branch name in the bottom-left corner → "Create new branch").
  3. Edit the file(s) you need to, same as you would have in vi.
  4. Open the Source Control panel (the branching icon in the left sidebar). Review the changed files, stage them, write a short commit message, and click the checkmark to commit.
  5. Click Sync Changes (or git push) to send your branch to GitHub.
  6. On github.com, open a Pull Request from your branch, and merge it once it looks right.
What's happening: This loop — pull, branch, edit, commit, push, PR — is the entire replacement for "runs everything from Fivetran." Nothing you change is live until you push it and merge the PR; until then it only exists on your machine and your branch, so there's no way to break the shared version by editing locally.
Reference
  • Quickstart: use source control in VS Code
  • Collaborating with pull requests — GitHub Docs
